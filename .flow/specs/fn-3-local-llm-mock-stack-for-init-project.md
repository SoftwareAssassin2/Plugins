# Local LLM mock stack for /init-project (LiteLLM + Ollama)

## Overview

Adds an **opt-in local LLM mock stack** to the `/init-project` scaffolding (epic **fn-2**, `fn-2-init-project-scaffolding-skill-init`). When a developer runs a scaffolded project locally, this stack lets the app's outbound LLM calls hit a **local gateway** instead of paid/cloud providers.

**Topology (locked via interview):**

```
app (OpenAI SDK)    ─┐
                     ├─►  LiteLLM Proxy  ──►  Ollama  ──►  model
app (Anthropic SDK) ─┘    :4000                :11434
                          /v1/chat/completions  (OpenAI surface)
                          /v1/messages          (Anthropic surface)
```

- **LiteLLM Proxy** = the single gateway. Hosts BOTH the OpenAI-compatible surface (`/v1/chat/completions`) AND the Anthropic-compatible surface (the **unified `/v1/messages`** endpoint — NOT the raw `/anthropic` passthrough, which forwards to real api.anthropic.com). Translates both formats to one backend.
- **Ollama** = the inference engine (real models, GGUF/llama.cpp, official Docker image), exposed only to LiteLLM. The app never talks to Ollama directly.
- No local engine (Ollama / LM Studio / llama.cpp) speaks Anthropic format natively, so LiteLLM is required for `/v1/messages`. Ollama is chosen over LM Studio because it is container-native (LM Studio is GUI-first with no official image).

**Two operating modes (layered) — selected by compose profile:**
- **Deterministic mock** (`--profile ai-mock`) — LiteLLM only, mounting `config.mock.yaml` whose models carry `mock_response`; **Ollama is not started**. For CI/tests: fast, free, deterministic.
- **Real local inference** (`--profile ai`) — LiteLLM + Ollama + model-pull init, mounting the real `config.yaml`. For manual dev/demos. Switchable per-environment without editing app code (the app's base URL is identical for both; only the active profile differs).

**Config invariant (resolved in review):** `services.claude-api` AND `services.openai-api` are **always** part of the scaffolded config model (like fn-2's existing `claude-api`), each carrying explicit `base_url` + `api_key` fields (never omitted), so `build-config` never branches on the opt-in. **Concrete fields/values:** non-opt-in → `claude-api.base_url = https://api.anthropic.com`, `openai-api.base_url = https://api.openai.com/v1`, both `api_key = REPLACE_ME`. Opt-in (local) → `claude-api.base_url = http://127.0.0.1:4000` (Anthropic SDK root), `openai-api.base_url = http://127.0.0.1:4000/v1` (OpenAI), both `api_key = sk-local-mock` (dummy non-empty — SDKs reject empty keys; LiteLLM accepts any when no master key is set). What the local-LLM opt-in changes is only: (a) those local `config.json` base URLs + keys, (b) the `etc/local-llm/` stack templates are laid down, (c) `config.json` gains `localLlm.model` (the chosen model). Without opt-in, no `etc/local-llm/` and no `localLlm` block exist.

**Model-name validation:** `localLlm.model` is user-supplied (incl. via the "something else" path), so it is validated against the Ollama model-name grammar `^[A-Za-z0-9._/-]+(:[A-Za-z0-9._-]+)?$` **at scaffold selection time (R4) AND in `build-config` before it is stamped into the LiteLLM `config.yaml` (R6)** — a hostile value with whitespace, shell metacharacters, or YAML-sensitive characters cannot corrupt the stamped `config.yaml` or the exported `LLM_MODEL`.

**Model source of truth (resolved in review):** the chosen model lives at `config.json` → `localLlm.model` (present only on opt-in), the single authoritative source. Two consumers read it, both from `config.json` (no second generated copy): (1) `build-config` stamps the **gitignored** `etc/local-llm/litellm/config.yaml` from a committed `config.yaml.template` whose model line is `ollama_chat/@@LLM_MODEL@@` — replacing the `@@LLM_MODEL@@` token with the **raw validated model name** (never re-prefixing `ollama_chat/`); (2) `system.sh up`/`down` **export `LLM_MODEL` (from `config.json` via `jq`, default empty)** into the environment before invoking `docker compose`, and the compose model-pull service interpolates `${LLM_MODEL:-}` into the container env (`ollama pull "$LLM_MODEL"`). **No generated `.env` file and no `env_file:` directive** — this avoids the `env_file: {required:false}` compose-version dependency and keeps `down` working with no generated artifacts. **No hardcoded default model lives in committed templates.**

**Classification:** this is **internal dev tooling** living under `etc/local-llm/`, mirroring the Grafana/OTel observability stack (fn-2 R5). It is **explicitly NOT a `systems[]` component** (it exists only in local dev — nothing deploys it) and **NOT itself a `services{}` external dependency**. It *backs* the `claude-api` + `openai-api` `services{}` entries in local dev.

This is a **new epic that depends on fn-2** and extends fn-2-authored, build-time-complete templates (currently in worktree `.worktrees/init-project/plugins/init-project/templates/`): `config.json` / `config.deploy.json`, `src/system-cli/system.sh` / `up.sh` / `down.sh` / `build-config.sh`, `_CLAUDE.md`, `docs/dev-container.md`, `docs/config-management.md`, `README.md`, plus the scaffold engine + `SKILL.md`.

## Quick commands
```bash
# Stack templates present and behind opt-in profiles (real + mock)
test -f etc/local-llm/docker-compose.yml
grep -Eq 'profiles:|ai-mock' etc/local-llm/docker-compose.yml

# AUTOMATED (CI gate): generated-file checks + mocked shell — NO docker/daemon/model
./system.sh build-config && test -f src/Api/.env && for v in ANTHROPIC_BASE_URL ANTHROPIC_API_KEY OPENAI_BASE_URL OPENAI_API_KEY; do grep -q "^$v=" src/Api/.env; done
bash tests/system-cli/*.sh   # kcov-covered, ollama/curl/docker stubbed

# Deterministic mock mode answers with NO ollama container (CI-friendly)
./system.sh up --profile ai-mock   # (raw `docker compose --profile ai-mock ...` is a lower-level diagnostic only)
curl -fsS localhost:4000/v1/messages -H 'content-type: application/json' -H 'anthropic-version: 2023-06-01' \
  -d '{"model":"local","max_tokens":16,"messages":[{"role":"user","content":"ping"}]}' | jq -e '.content[0].text'

# MANUAL/INTEGRATION proof (real inference; requires `build-config` first to stamp the model; pulls a model; not a CI gate)
./system.sh build-config && ./system.sh up --profile ai
curl -fsS localhost:4000/v1/chat/completions -H 'content-type: application/json' \
  -d '{"model":"local","messages":[{"role":"user","content":"ping"}]}' | jq -e '.choices[0].message.content'

# Default `up` excludes the stack; opt-in activates a profile
./system.sh up               # no litellm/ollama containers
./system.sh up --profile ai-mock   # litellm (mock) only
./system.sh up --profile ai        # litellm + ollama + model-pull
```

## Boundaries / non-goals
- NOT a deployed/production component — local-dev mock only. `config.deploy.json` points `claude-api`/`openai-api` at the real providers; nothing deploys LiteLLM or Ollama.
- Does NOT change fn-2's existing components (postgres, keycloak, the .NET/Angular starter, observability) beyond adding the `openai-api` `services{}` entry, the new compose stack, and the dispatcher/doc wiring.
- Does NOT add a LiteLLM MCP server or new devcontainer CLI tool — docker-in-docker + `jq` from fn-2 R4 already suffice.
- Does NOT guarantee high-fidelity Anthropic tool-use/streaming through real Ollama inference (see R3/R11 caveats) — deterministic mock is the recommended path for tool-use test assertions.
- Model weights are not committed; Ollama pulls the chosen model on first boot.
- The **real-inference** smoke (docker + LiteLLM + Ollama + a pulled model) is **manual/integration evidence, not part of the automated CI gate**; CI exercises generated files + mocked shell only (R10).

## Decision context
- **`etc/local-llm/`, not `systems[]`:** the stack is the same *kind* as observability (fn-2 R5) — compose-managed internal tooling, outside `.devcontainer/`, orchestrated by `system.sh up`, outside the `src/<component>` ↔ `systems[]` invariant. Rejected the "give LiteLLM/Ollama their own `src/` + `systems[]` entries" framing: postgres/keycloak are real deployed components; this mock has no deploy target.
- **`openai-api` always present; only repoint + stack are conditional:** keeping both services unconditional avoids a `build-config` branch and a config-model fork (the review flagged the original contradiction). The opt-in toggles base URLs + lays down `etc/local-llm/`.
- **Two operating modes via two profiles + two configs:** `config.yaml` (real, `ollama_chat` backend) under `--profile ai`; `config.mock.yaml` (`mock_response`, no backend) under `--profile ai-mock` (LiteLLM only, Ollama absent). The app's base URL is identical across modes; only the active profile changes — so no app code edits to switch. CI selects `ai-mock`.
- **LiteLLM → Ollama via native `ollama_chat/<model>`** (not `openai/` + Ollama's `/v1`): Ollama's OpenAI-compat surface is officially experimental and unreliable for tool calls. Also set `OLLAMA_API_BASE` env on the LiteLLM container — LiteLLM hits `OLLAMA_API_BASE/api/show` for capability detection regardless of per-model `api_base` (known bug), and without it function-calling capability silently degrades.
- **Unified `/v1/messages`, pinned image:** the proxy serves `/v1/messages` automatically for every `model_list` entry, translating Anthropic↔OpenAI. The OpenAI↔Anthropic SSE re-assembly has documented regressions (dropped streaming tool_use args, null `stop_reason`, dropped first text delta) that are version-specific — so **pin the LiteLLM image to a specific immutable version tag (or digest)** and record the tested version; do NOT use `latest`/`main-latest`/a bare moving channel tag.
- **Model-pull init container:** uses the **same `ollama/ollama` image** (so the `ollama` CLI is present — do NOT assume `curl` exists in images), with `OLLAMA_HOST=http://ollama:11434` and command `ollama pull "$MODEL"`; `restart: "no"`; gated by `depends_on: ollama` `condition: service_healthy`. The `ollama` service healthcheck uses `ollama list` (CLI), not `curl`.
- **Deterministic mock via `mock_response`:** short-circuits the backend (Ollama need not run) — the CI/test path. Recommended for tool-use assertions given the streaming-fidelity gaps.
- **build-config contract (explicit):** `build-config` reads the always-present `services.claude-api.{base_url,api_key}` + `services.openai-api.{base_url,api_key}` and writes the LLM service env into the **consuming backend component's `.env` (the `Api` by default)** as `ANTHROPIC_BASE_URL` + `ANTHROPIC_API_KEY` and `OPENAI_BASE_URL` + `OPENAI_API_KEY` (SDK-standard names). `config.deploy.json` supplies `{{CLAUDE_API_BASE_URL}}`/`{{CLAUDE_API_KEY}}`/`{{OPENAI_API_BASE_URL}}`/`{{OPENAI_API_KEY}}`. External creds stay opaque (exempt from URL-safe validation). When `localLlm.model` is present, `build-config` ALSO stamps the gitignored `etc/local-llm/litellm/config.yaml` (from `config.yaml.template`); it writes **no** `etc/local-llm/.env` (the model-pull `LLM_MODEL` is exported by `system.sh up`/`down` from `config.json`, not a file). Each emitted component-`.env` line is asserted by a test.
- **`down` semantics (resolved in review):** `up` is profile-gated (default `up` starts no AI containers; `--profile ai`/`ai-mock` activate them). Default `system.sh down` **tears down `etc/local-llm/` if the compose file is present** — invoking `docker compose ... --profile ai --profile ai-mock down` so nothing started under either profile is left running — and starts nothing. The `[[ -f ]]` guard makes a not-installed stack a no-op. This asymmetry (down is not profile-gated, up is) is explicit and tested. **`down` must succeed post-clone, pre-`build-config`** (the generated `litellm/config.yaml` absent): `LLM_MODEL` reaches compose via a `system.sh`-exported env var interpolated as `${LLM_MODEL:-}` (NO `env_file:`, so no compose-version dependency and nothing to miss); the real LiteLLM service mounts its generated `litellm/config.yaml` via a **bind mount (`volumes:`) with `create_host_path: false`, NOT a top-level `configs:` with `file:`** (the latter is validated at parse time and would make `down` fail when the generated file is absent). `_pull-model.sh` fails clearly when `LLM_MODEL` is empty. Tests cover `system.sh down` with the compose file present but `litellm/config.yaml` absent.
- **Model menu on opt-in (R4):** lightweight (`llama3.2:3b` / `qwen2.5:3b`, laptop/CPU-friendly default), more powerful (`qwen2.5:7b` / `llama3.1:8b`), abliterated (`huihui_ai/llama3.2-abliterate`, `huihui_ai/qwen2.5-abliterate:7b-instruct`), and "something else" (skill guides selection by size/VRAM/task/license). Abliterated carries a quality-degradation + removed-guardrails + base-model-license caveat; never a default-on. Chosen model written to `config.json`; Ollama pulls on first boot.
- **Build-time-complete (fn-2 R23):** every new template file is authored to final form in this epic; the scaffold engine only copies + applies `__SCAFFOLD_*__` substitution (the chosen model name) and conditional inclusion on opt-in. External API keys stay `REPLACE_ME`.
- **Dispatcher arg forwarding (R7):** `system.sh` must forward `"$@"` to the subcommand and `up.sh`/`down.sh` must parse `--profile <name>` — the current `up.sh` ignores args, so this is an explicit in-scope change (not assumed). `up.sh` guards stacks with `[[ -f "$stack" ]]`, so a not-installed stack is a no-op.
- **Compose profile gotchas (R7):** a profiled service is NOT auto-started by another service's `depends_on` unless its profile is active; profiled services still need a shared network or DNS-by-name fails silently; pair `depends_on` with `condition: service_healthy`.

## Acceptance Criteria
- **R1:** A new compose stack `etc/local-llm/docker-compose.yml` defines `ollama` (official `ollama/ollama`, named volume at `/root/.ollama`, CPU-only by default, healthcheck via `ollama list` with a generous `start_period` — NOT curl), `litellm` (pinned by **image digest** — `image: ghcr.io/berriai/litellm@sha256:…`, optionally with a readable version tag alongside; a bare semver tag is insufficient since registry tags can be re-pushed — the exact tested version+digest captured in fn-3….1's task evidence for fn-3….5 to document; config-mounted, port 4000), and a **model-pull init container** (same `ollama/ollama` image with an **`entrypoint:` override** so the mounted `_pull-model.sh` actually executes — the official image's default entrypoint is `ollama`, so a bare `command:` would become `ollama _pull-model.sh`; `OLLAMA_HOST=http://ollama:11434`, `LLM_MODEL` from compose `${LLM_MODEL:-}` interpolation, `ollama pull "$LLM_MODEL"`, `restart: "no"`, gated by `depends_on: ollama service_healthy`). The **real `litellm` waits for the pull**: `depends_on: model-pull condition: service_completed_successfully` (so early smoke requests don't hit a missing model). Real services tagged `profiles: ["ai"]`; the mock LiteLLM tagged `profiles: ["ai-mock"]`; all on a shared network; **LiteLLM published on loopback only — `127.0.0.1:4000:4000`** (a dev-only unauthenticated gateway must not bind all interfaces; matches the `127.0.0.1` SDK base URLs), Ollama internal-only (no host port). Authored build-time-complete.
- **R2:** LiteLLM routes to Ollama via the native **`ollama_chat/<model>`** provider with `api_base: http://ollama:11434` AND the `OLLAMA_API_BASE` env set on the container; it serves the OpenAI **`/v1/chat/completions`** surface AND the Anthropic **unified `/v1/messages`** surface (not the raw `/anthropic` passthrough). Manual/integration smoke: both surfaces return a completion against the pulled model.
- **R3:** A **deterministic mock mode** (LiteLLM `config.mock.yaml` with `mock_response`) runs under `--profile ai-mock` with **no Ollama container started**, returning canned output through **BOTH** the OpenAI `/v1/chat/completions` AND the Anthropic `/v1/messages` surfaces (verified by the optional Docker-enabled integration job in R10 — LiteLLM only, no model; distinct from the no-daemon unit gate). Mode is selected by compose profile only — no app-code edits, identical base URL. CI uses `ai-mock`. The Anthropic-surface tool-use/streaming fidelity gaps are documented, and mock mode is recommended for deterministic tool-use test assertions.
- **R4:** The model is chosen **interactively at scaffold time, only when the user opts into installing the stack**. The menu offers: lightweight (e.g. `llama3.2:3b`), more powerful (e.g. `qwen2.5:7b`), abliterated (e.g. `huihui_ai/llama3.2-abliterate`, with a quality/guardrail/license caveat), and **"something else"** (skill guides selection by size/VRAM/task/license). The chosen model name is written to **`config.json` → `localLlm.model`** (the single source of truth); `build-config` stamps it into the gitignored `litellm/config.yaml` (`ollama_chat/@@LLM_MODEL@@` → raw model), and `system.sh up` exports `LLM_MODEL` from `config.json` so the model-pull container pulls it on first boot (no generated `.env`). The chosen/entered model (incl. the "something else" path) is validated against `^[A-Za-z0-9._/-]+(:[A-Za-z0-9._-]+)?$` at selection time. No hardcoded default model lives in committed templates.
- **R5:** The `etc/local-llm/` stack + the local base-URL repoint are **opt-in and removable**: not laid down / not repointed unless the user opts in at scaffold time; even when installed, the stack sits behind the `ai`/`ai-mock` compose profiles and is **excluded from the default `system.sh up`**. (`services.claude-api`/`services.openai-api` themselves are always present — see R6.) Marked removable like `SampleApp`.
- **R6:** `config.json` and `config.deploy.json` **always** carry both `claude-api` and `openai-api` `services{}` entries, each with explicit `base_url` + `api_key` fields (key-set parity). Non-opt-in local values: `claude-api.base_url=https://api.anthropic.com`, `openai-api.base_url=https://api.openai.com/v1`, keys `REPLACE_ME`. Opt-in local values: `claude-api.base_url=http://127.0.0.1:4000`, `openai-api.base_url=http://127.0.0.1:4000/v1`, keys `sk-local-mock`. `config.deploy.json` uses `{{CLAUDE_API_BASE_URL}}`/`{{CLAUDE_API_KEY}}`/`{{OPENAI_API_BASE_URL}}`/`{{OPENAI_API_KEY}}` placeholders. `build-config` (unconditionally, no opt-in branch) distributes `ANTHROPIC_BASE_URL`/`ANTHROPIC_API_KEY` + `OPENAI_BASE_URL`/`OPENAI_API_KEY` into the consuming component's `.env` (the `Api` by default); external creds exempt from URL-safe validation. `build-config` **validates each `base_url`** against a concrete service-base-url grammar — a restrictive host class (`[A-Za-z0-9.-]+` — no IPv6 needed; `?#@[]:` and other non-host chars are rejected **in the host portion**): `^https?://[A-Za-z0-9.-]+(:[0-9]{1,5})?(/[^[:space:]]*)?$` (an optional path may follow — query/fragment in a path are permitted, since a legitimate base URL can carry a path; the validation targets a well-formed scheme+host+port, not full RFC-3986 path purity), AND the captured port (if present) is range-checked **1–65535 separately** (the `[0-9]{1,5}` class alone would accept `99999`) — stricter than the existing permissive `v_url`, with positive/negative fixtures (incl. invalid + non-numeric ports); only the `api_key` fields stay opaque/unvalidated. **Opt-in invariant:** if `localLlm.model` is present but `etc/local-llm/litellm/config.yaml.template` is missing, `build-config` **fails with a clear error** (rather than silently skipping). `api_key` values stay semantically opaque (exempt from URL-safe validation) but are still made **transport-safe**: `build-config` rejects CR/LF/control chars (or dotenv-quotes) so an opaque key cannot corrupt `.env` or inject extra variables — tested with a non-URL-safe-but-line-safe key. When stamping `litellm/config.yaml`, `build-config` validates `localLlm.model` against `^[A-Za-z0-9._/-]+(:[A-Za-z0-9._-]+)?$` before deterministic replacement of the named token `@@LLM_MODEL@@` (with the raw model name) in `config.yaml.template` — failing on an invalid model rather than emitting corrupt YAML. A test asserts each emitted component-`.env` line, base_url validation (reject malformed), model-name validation (reject hostile; accept valid `/` and `:` punctuation), the missing-template failure, and that raw `config.deploy.json` still fails the unrendered-placeholder guard while a rendered deploy config distributes the vars. The `config-management.md` schema table gains the `openai-api` row + the new env vars **AND documents the opt-in-only top-level `localLlm.model` field** (its validation grammar + that it drives the generated `config.yaml`/exported `LLM_MODEL`).
- **R7:** `system.sh` forwards `"$@"` to subcommands and `up.sh` parses profiles with an explicit grammar: accepts only **`ai`** and **`ai-mock`**, via repeatable `--profile <name>` and `--profile=<name>` forms; **rejects unknown profiles/flags — and rejects selecting BOTH `ai` and `ai-mock` together (both LiteLLM instances publish `:4000`, a port conflict) — with usage exit 64**; passes only the accepted profile through to the `etc/local-llm/` compose invocation. (`down` is exempt — it passes both profiles internally to guarantee teardown, never starting anything.) `system.sh up --profile ai` (or `ai-mock`) brings up the `etc/local-llm/` stack alongside postgres/keycloak/observability, while the default `up` starts none of it. **`down` is NOT profile-gated:** default `system.sh down` tears down `etc/local-llm/` if the compose file is present (passing `--profile ai --profile ai-mock` so nothing started under either profile is left running) and starts nothing — this up/down asymmetry is explicit and tested. The `[[ -f "$stack" ]]` guard + `# Description:` comments are updated. Compose profile/`depends_on`/network/health gotchas handled (profiled service not auto-started via `depends_on`; shared network; `condition: service_healthy` on the model-pull gate).
- **R8:** A new **`docs/local-llm.md`** standard is authored (what/why, opt-in install, `ai` vs `ai-mock` profiles, mock vs real modes, model selection incl. the abliterated caveat, the Anthropic-surface fidelity caveat + pinned-version rationale, and **complete removal instructions** — to fully remove the mock a developer must (1) delete `etc/local-llm/`, (2) remove the `localLlm` block from `config.json`, and (3) restore `services.claude-api`/`services.openai-api` `base_url`+`api_key` to real-provider values, then rerun `build-config`; doing only (1) leaves `build-config` failing on the orphaned `localLlm.model` and the app pointed at a missing gateway. The `[[ -f ]]` guard only covers the `up`/`down` no-op, not full removal) and indexed in the `_CLAUDE.md` Standards index. `docs/dev-container.md`'s dep-placement example and `_CLAUDE.md`'s invariant-exceptions table are extended to list `etc/local-llm/`. `README.md` gains a "Local LLM mock (opt-in)" section.
- **R9:** The stack is documented as **internal dev tooling — explicitly NOT a `systems[]` component and NOT itself a `services{}` external dependency** (it backs the `claude-api`/`openai-api` services in local dev), mirroring the observability classification.
- **R10:** Two test tiers, kept distinct: **(a) the unit gate** validates generated files (component-`.env` lines, presence/shape of compose + config templates) and **shell branch coverage with `ollama`/`curl`/`docker` stubbed — no real network, model, or daemon** — 100% line coverage via kcov + a test per branch for all new/changed shell (model-pull `_pull-model.sh`, `system.sh`/`up.sh`/`down.sh`/`build-config.sh` additions, and the `scaffold.sh` opt-in/model-flag changes). **(b) an optional Docker-enabled integration job** brings up `--profile ai-mock` (LiteLLM only, no model — fast/cheap) and asserts canned responses on both `/v1/chat/completions` and `/v1/messages`. The **real-inference** `--profile ai` smoke (R2) is **manual/integration evidence, never the unit gate**. Compose YAML and the LiteLLM configs are config, not code.
- **R11:** The LiteLLM image is **pinned by digest** (`@sha256:…`, optionally with a readable version tag alongside), never `latest`/`main-latest`/a bare moving tag (registry tags can be re-pushed), because of documented `/v1/messages` streaming-translation regressions; the exact tested version + digest is recorded in `docs/local-llm.md`.

## Early proof point
Task **fn-3-local-llm-mock-stack-for-init-project.1** stands up the `etc/local-llm/` compose stack and proves (manually/integration) that both LiteLLM surfaces (`/v1/chat/completions` + unified `/v1/messages`) answer against a real Ollama model under `--profile ai`, AND that `--profile ai-mock` answers via `mock_response` with no Ollama container. If that loop can't be made to work (especially the Anthropic surface), reconsider the LiteLLM-as-gateway approach before investing in the config/dispatcher/docs wiring.
