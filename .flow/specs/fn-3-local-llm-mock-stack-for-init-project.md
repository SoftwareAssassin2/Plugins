# Local LLM mock stack for /init-project (LiteLLM + Ollama)

## Overview

Adds an **opt-in local LLM mock stack** to the `/init-project` scaffolding (epic **fn-2**, `fn-2-init-project-scaffolding-skill-init`). When a developer runs a scaffolded project locally, this stack lets the app's outbound LLM calls hit a **local gateway** instead of paid/cloud providers.

**Topology (locked via interview):**

```
app (OpenAI SDK)    ‚îÄ‚îê
                     ‚îú‚îÄ‚ñ∫  LiteLLM Proxy  ‚îÄ‚îÄ‚ñ∫  Ollama  ‚îÄ‚îÄ‚ñ∫  model
app (Anthropic SDK) ‚îÄ‚îò    :4000                :11434
                          /v1/chat/completions  (OpenAI surface)
                          /v1/messages          (Anthropic surface)
```

- **LiteLLM Proxy** = the single gateway. Hosts BOTH the OpenAI-compatible surface (`/v1/chat/completions`) AND the Anthropic-compatible surface (the **unified `/v1/messages`** endpoint ‚Äî NOT the raw `/anthropic` passthrough, which forwards to real api.anthropic.com). Translates both formats to one backend.
- **Ollama** = the inference engine (real models, GGUF/llama.cpp, official Docker image), exposed only to LiteLLM. The app never talks to Ollama directly.
- No local engine (Ollama / LM Studio / llama.cpp) speaks Anthropic format natively, so LiteLLM is required for `/v1/messages`. Ollama is chosen over LM Studio because it is container-native (LM Studio is GUI-first with no official image).

**Two operating modes (layered) ‚Äî selected by compose profile:**
- **Deterministic mock** (`--profile ai-mock`) ‚Äî LiteLLM only, mounting `config.mock.yaml` whose models carry `mock_response`; **Ollama is not started**. For CI/tests: fast, free, deterministic.
- **Real local inference** (`--profile ai`) ‚Äî LiteLLM + Ollama + model-pull init, mounting the real `config.yaml`. For manual dev/demos. Switchable per-environment without editing app code (the app's base URL is identical for both; only the active profile differs).

**Config invariant (resolved in review):** `services.claude-api` AND `services.openai-api` are **always** part of the scaffolded config model (like fn-2's existing `claude-api`), each carrying explicit `base_url` + `api_key` fields (never omitted), so `build-config` never branches on the opt-in. **Concrete fields/values:** non-opt-in ‚Üí `claude-api.base_url = https://api.anthropic.com`, `openai-api.base_url = https://api.openai.com/v1`, both `api_key = REPLACE_ME`. Opt-in (local) ‚Üí `claude-api.base_url = http://127.0.0.1:4000` (Anthropic SDK root), `openai-api.base_url = http://127.0.0.1:4000/v1` (OpenAI), both `api_key = sk-local-mock` (dummy non-empty ‚Äî SDKs reject empty keys; LiteLLM accepts any when no master key is set). What the local-LLM opt-in changes is only: (a) those local `config.json` base URLs + keys, (b) the `etc/local-llm/` stack templates are laid down, (c) `config.json` gains `localLlm.model` (the chosen model). Without opt-in, no `etc/local-llm/` and no `localLlm` block exist.

**Model-name validation:** `localLlm.model` is user-supplied (incl. via the "something else" path), so it is validated against the Ollama model-name grammar `^[A-Za-z0-9._/-]+(:[A-Za-z0-9._-]+)?$` **at scaffold selection time (R4) AND in `build-config` before it is stamped into the LiteLLM `config.yaml` (R6)** ‚Äî a hostile value with whitespace, shell metacharacters, or YAML-sensitive characters cannot corrupt the stamped `config.yaml` or the exported `LLM_MODEL`.

**Model source of truth (resolved in review):** the chosen model lives at `config.json` ‚Üí `localLlm.model` (present only on opt-in), the single authoritative source. Two consumers read it, both from `config.json` (no second generated copy): (1) `build-config` stamps the **gitignored** `etc/local-llm/litellm/config.yaml` from a committed `config.yaml.template` whose model line is `ollama_chat/@@LLM_MODEL@@` ‚Äî replacing the `@@LLM_MODEL@@` token with the **raw validated model name** (never re-prefixing `ollama_chat/`); (2) `system.sh up`/`down` **export `LLM_MODEL` (from `config.json` via `jq`, default empty)** into the environment before invoking `docker compose`, and the compose model-pull service interpolates `${LLM_MODEL:-}` into the container env (`ollama pull "$LLM_MODEL"`). **No generated `.env` file and no `env_file:` directive** ‚Äî this avoids the `env_file: {required:false}` compose-version dependency and keeps `down` working with no generated artifacts. **No hardcoded default model lives in committed templates.**

**Classification:** this is **internal dev tooling** living under `etc/local-llm/`, mirroring the Grafana/OTel observability stack (fn-2 R5). It is **explicitly NOT a `systems[]` component** (it exists only in local dev ‚Äî nothing deploys it) and **NOT itself a `services{}` external dependency**. It *backs* the `claude-api` + `openai-api` `services{}` entries in local dev.

This is a **new epic that depends on fn-2** and extends fn-2-authored, build-time-complete templates (currently in worktree `plugins/init-project/templates/`): `config.json` / `config.deploy.json`, `src/system-cli/system.sh` / `up.sh` / `down.sh` / `build-config.sh`, `_CLAUDE.md`, `docs/dev-container.md`, `docs/config-management.md`, `README.md`, plus the scaffold engine + `SKILL.md`.

## Quick commands
```bash
# Stack templates present and behind opt-in profiles (real + mock)
test -f etc/local-llm/docker-compose.yml
grep -Eq 'profiles:|ai-mock' etc/local-llm/docker-compose.yml

# AUTOMATED (CI gate): generated-file checks + mocked shell ‚Äî NO docker/daemon/model
./system.sh build-config && test -f src/Api/.env && for v in ANTHROPIC_BASE_URL ANTHROPIC_API_KEY OPENAI_BASE_URL OPENAI_API_KEY; do grep -q "^$v=" src/Api/.env; done
bash tests/system-cli/*.sh   # kcov-covered, ollama/curl/docker stubbed

# Deterministic mock mode answers with NO ollama container (CI-friendly)
./system.sh up --profile ai-mock   # (raw `docker compose --profile ai-mock ...` is a lower-level diagnostic only)
curl -fsS localhost:4000/v1/messages -H 'content-type: application/json' -H 'anthropic-version: 2023-06-01' \
  -d '{"model":"local","max_tokens":16,"messages":[{"role":"user","content":"ping"}]}' | jq -e '.content[0].text'

# MANUAL/INTEGRATION proof (real inference; requires `build-config` first to stamp the model; pulls a model; not a CI gate)
./system.sh build-config && ./system.sh up --profile ai
curl -fsS localhost:4000/v1/chat/completions -H 'content-type: application/json' \
  -d '{"model":"local","messages":[{"role":"user","content":"ping"}]}' | jq -e '.choices[0].message.content'

# Default `up` excludes the stack; opt-in activates a profile
./system.sh up               # no litellm/ollama containers
./system.sh up --profile ai-mock   # litellm (mock) only
./system.sh up --profile ai        # litellm + ollama + model-pull
```

## Boundaries / non-goals
- NOT a deployed/production component ‚Äî local-dev mock only. `config.deploy.json` points `claude-api`/`openai-api` at the real providers; nothing deploys LiteLLM or Ollama.
- Does NOT change fn-2's existing components (postgres, keycloak, the .NET/Angular starter, observability) beyond adding the `openai-api` `services{}` entry, the new compose stack, and the dispatcher/doc wiring.
- Does NOT add a LiteLLM MCP server or new devcontainer CLI tool ‚Äî docker-in-docker + `jq` from fn-2 R4 already suffice.
- Does NOT guarantee high-fidelity Anthropic tool-use/streaming through real Ollama inference (see R3/R11 caveats) ‚Äî deterministic mock is the recommended path for tool-use test assertions.
- Model weights are not committed; Ollama pulls the chosen model on first boot.
- The **real-inference** smoke (docker + LiteLLM + Ollama + a pulled model) is **manual/integration evidence, not part of the automated CI gate**; CI exercises generated files + mocked shell only (R10).

## Decision context
- **`etc/local-llm/`, not `systems[]`:** the stack is the same *kind* as observability (fn-2 R5) ‚Äî compose-managed internal tooling, outside `.devcontainer/`, orchestrated by `system.sh up`, outside the `src/<component>` ‚Üî `systems[]` invariant. Rejected the "give LiteLLM/Ollama their own `src/` + `systems[]` entries" framing: postgres/keycloak are real deployed components; this mock has no deploy target.
- **`openai-api` always present; only repoint + stack are conditional:** keeping both services unconditional avoids a `build-config` branch and a config-model fork (the review flagged the original contradiction). The opt-in toggles base URLs + lays down `etc/local-llm/`.
- **Two operating modes via two profiles + two configs:** `config.yaml` (real, `ollama_chat` backend) under `--profile ai`; `config.mock.yaml` (`mock_response`, no backend) under `--profile ai-mock` (LiteLLM only, Ollama absent). The app's base URL is identical across modes; only the active profile changes ‚Äî so no app code edits to switch. CI selects `ai-mock`. The mock uses **a single canned `mock_response`** valid for both chat surfaces, editable by hand in `config.mock.yaml` (documented); per-model/per-endpoint mocking is a deliberate non-goal for v1 (`ai-mock` is chat-only ‚Äî no embeddings mock).
- **LiteLLM ‚Üí Ollama via native `ollama_chat/<model>`** (not `openai/` + Ollama's `/v1`): Ollama's OpenAI-compat surface is officially experimental and unreliable for tool calls. Also set `OLLAMA_API_BASE` env on the LiteLLM container ‚Äî LiteLLM hits `OLLAMA_API_BASE/api/show` for capability detection regardless of per-model `api_base` (known bug), and without it function-calling capability silently degrades.
- **Unified `/v1/messages`, pinned image:** the proxy serves `/v1/messages` automatically for every `model_list` entry, translating Anthropic‚ÜîOpenAI. The OpenAI‚ÜîAnthropic SSE re-assembly has documented regressions (dropped streaming tool_use args, null `stop_reason`, dropped first text delta) that are version-specific ‚Äî so **pin the LiteLLM image to a specific immutable version tag (or digest)** and record the tested version; do NOT use `latest`/`main-latest`/a bare moving channel tag.
- **Model-pull init container:** uses the **same `ollama/ollama` image** (so the `ollama` CLI is present ‚Äî do NOT assume `curl` exists in images), with `OLLAMA_HOST=http://ollama:11434` and command `ollama pull "$MODEL"`; `restart: "no"`; gated by `depends_on: ollama` `condition: service_healthy`. The `ollama` service healthcheck uses `ollama list` (CLI), not `curl`.
- **Deterministic mock via `mock_response`:** short-circuits the backend (Ollama need not run) ‚Äî the CI/test path. Recommended for tool-use assertions given the streaming-fidelity gaps.
- **build-config contract (explicit):** `build-config` reads the always-present `services.claude-api.{base_url,api_key}` + `services.openai-api.{base_url,api_key}` and writes the LLM service env into the **consuming backend component's `.env` (the `Api` by default)** as `ANTHROPIC_BASE_URL` + `ANTHROPIC_API_KEY` and `OPENAI_BASE_URL` + `OPENAI_API_KEY` (SDK-standard names). `config.deploy.json` supplies `{{CLAUDE_API_BASE_URL}}`/`{{CLAUDE_API_KEY}}`/`{{OPENAI_API_BASE_URL}}`/`{{OPENAI_API_KEY}}`. External creds stay opaque (exempt from URL-safe validation). When `localLlm.model` is present, `build-config` ALSO stamps the gitignored `etc/local-llm/litellm/config.yaml` (from `config.yaml.template`); it writes **no** `etc/local-llm/.env` (the model-pull `LLM_MODEL` is exported by `system.sh up`/`down` from `config.json`, not a file). Each emitted component-`.env` line is asserted by a test.
- **`down` semantics (resolved in review):** `up` is profile-gated (default `up` starts no AI containers; `--profile ai`/`ai-mock` activate them). Default `system.sh down` **tears down `etc/local-llm/` if the compose file is present** ‚Äî invoking `docker compose ... --profile ai --profile ai-mock down` so nothing started under either profile is left running ‚Äî and starts nothing. The `[[ -f ]]` guard makes a not-installed stack a no-op. This asymmetry (down is not profile-gated, up is) is explicit and tested. **`down` must succeed post-clone, pre-`build-config`** (the generated `litellm/config.yaml` absent): `LLM_MODEL` reaches compose via a `system.sh`-exported env var interpolated as `${LLM_MODEL:-}` (NO `env_file:`, so no compose-version dependency and nothing to miss); the real LiteLLM service mounts its generated `litellm/config.yaml` via a **bind mount (`volumes:`) with `create_host_path: false`, NOT a top-level `configs:` with `file:`** (the latter is validated at parse time and would make `down` fail when the generated file is absent). `_pull-model.sh` fails clearly when `LLM_MODEL` is empty. Tests cover `system.sh down` with the compose file present but `litellm/config.yaml` absent.
- **Model menu on opt-in (R4):** lightweight (`llama3.2:3b` / `qwen2.5:3b`, laptop/CPU-friendly default), more powerful (`qwen2.5:7b` / `llama3.1:8b`), abliterated (`huihui_ai/llama3.2-abliterate`, `huihui_ai/qwen2.5-abliterate:7b-instruct`), and "something else" (skill guides selection by size/VRAM/task/license). Abliterated carries a quality-degradation + removed-guardrails + base-model-license caveat; never a default-on. Chosen model written to `config.json`; Ollama pulls on first boot.
- **Build-time-complete (fn-2 R23) + a sanctioned bounded exception:** every new template file is authored to final form in this epic; at runtime the engine copies + applies `__SCAFFOLD_*__` substitution and conditional file inclusion. The opt-in `config.json` edit (set `claude-api`/`openai-api` base URLs + keys, add `localLlm.model`/`embeddingModel`) is a **deterministic structured `jq` mutation of the generated project's `config.json`** ‚Äî an explicit, bounded extension of R23's "copy + substitute" rule, directly analogous to fn-2's sanctioned `__SCAFFOLD_GEN_URLSAFE__` secret-gen exception. It is NOT prose authoring, NOT applied to the committed template (only the generated copy), and touches only those named keys. External API keys stay `REPLACE_ME` unless opt-in sets the `sk-local-mock` dev value.
- **Dispatcher arg forwarding (R7):** `system.sh` must forward `"$@"` to the subcommand and `up.sh`/`down.sh` must parse `--profile <name>` ‚Äî the current `up.sh` ignores args, so this is an explicit in-scope change (not assumed). `up.sh` guards stacks with `[[ -f "$stack" ]]`, so a not-installed stack is a no-op.
- **Compose profile gotchas (R7):** a profiled service is NOT auto-started by another service's `depends_on` unless its profile is active; profiled services still need a shared network or DNS-by-name fails silently; pair `depends_on` with `condition: service_healthy`.
- **Embeddings (interview 2026-06-14, R12):** optional second model, opt-in via a separate setup prompt after the chat model. Stored at `config.json` ‚Üí `localLlm.embeddingModel` (optional). Served on the **OpenAI `/v1/embeddings` surface only** (Anthropic has no embeddings endpoint), via a second LiteLLM `model_list` entry (`local-embed` ‚Üí `ollama/<embeddingModel>`). Same single `ai` profile; the one model-pull step pulls both models. `ai-mock` stays chat-only (no embeddings mock). Default offer `nomic-embed-text` (+ "something else"). Rejected: bundling embeddings unconditionally (most mock cases are chat) and a separate embeddings profile (one `ai` profile is simpler).
- **GPU (interview 2026-06-14, R15):** CPU-only by default (laptop/CI portability). `docs/local-llm.md` documents a GPU compose-override snippet (`deploy.resources.reservations.devices`) as opt-in. No auto-detection.
- **Ollama pin (interview 2026-06-14, R14):** Ollama pinned to a specific **version tag** (not `latest`, not digest) ‚Äî it's the stable inference engine, less regression-prone than LiteLLM's translation layer (which is digest-pinned, R11), and a readable tag eases bumping.
- **Mock response shape (interview 2026-06-14):** a single editable canned `mock_response` (in `config.mock.yaml`), not per-model/per-endpoint ‚Äî keeps the deterministic CI path simple; richer fixtures are a documented power-user edit.
- **Tool-use/streaming stance (interview 2026-06-14, refined in review):** document the real-mode Anthropic-surface fidelity caveat AND have the Docker integration job (R3/R10) assert that a **tools-bearing request is tolerated (no 5xx, well-formed response)** on both surfaces ‚Äî NOT a structured tool-use round-trip (a canned `mock_response` can't emit `tool_calls`/`tool_use`). Structured tool-use translation fidelity is a documented MANUAL real-inference check, never asserted in CI.
- **Model-pull failure (interview 2026-06-14, R13):** a failed `ollama pull` (offline / unknown model) makes the model-pull init exit non-zero; because real `litellm` waits on `service_completed_successfully`, `up --profile ai` fails loudly with a clear message rather than leaving a half-started stack. Pre-pull/offline workflow documented. Rejected warn-and-continue (confusing late failures) and retry-with-backoff (complexity).
- **LLM consumer = `Api` only (interview 2026-06-14):** `build-config` writes the four LLM env vars only into `src/Api/.env` (the backend HTTP service boundary); `BusinessLogic`/`DataAccess` are libraries, not callers. Not configurable in v1.
- **Gateway keyless (interview 2026-06-14):** LiteLLM runs with NO master key locally ‚Äî it's loopback-only (`127.0.0.1:4000`), and the `sk-local-mock` value exists only because SDK clients reject empty keys. A generated master key was rejected as dev friction with no local threat.

## Acceptance Criteria
- **R1:** A new compose stack `etc/local-llm/docker-compose.yml` defines `ollama` (official `ollama/ollama` **pinned to a specific version tag, not `latest`** ‚Äî see R14; named volume at `/root/.ollama`, CPU-only by default ‚Äî see R15; healthcheck via `ollama list` with a generous `start_period` ‚Äî NOT curl), `litellm` (pinned by **image digest** ‚Äî `image: ghcr.io/berriai/litellm@sha256:‚Ä¶`, optionally with a readable version tag alongside; a bare semver tag is insufficient since registry tags can be re-pushed ‚Äî the exact tested version+digest captured in fn-3‚Ä¶.1's task evidence for fn-3‚Ä¶.5 to document; config-mounted, port 4000), and a **model-pull init container** (same `ollama/ollama` image with an **`entrypoint:` override** so the mounted `_pull-model.sh` actually executes ‚Äî the official image's default entrypoint is `ollama`, so a bare `command:` would become `ollama _pull-model.sh`; `OLLAMA_HOST=http://ollama:11434`, `LLM_MODEL` from compose `${LLM_MODEL:-}` interpolation, `ollama pull "$LLM_MODEL"`, `restart: "no"`, gated by `depends_on: ollama service_healthy`). The **real `litellm` waits for the pull**: `depends_on: model-pull condition: service_completed_successfully` (so early smoke requests don't hit a missing model). When **embeddings are opted in (R12)**, the single model-pull step pulls BOTH the chat model and the embedding model (still one `ai` profile). A failed `ollama pull` (offline / unknown model) makes model-pull exit non-zero, which blocks `litellm` and fails `up --profile ai` loudly (see R13). Real services tagged `profiles: ["ai"]`; the mock LiteLLM tagged `profiles: ["ai-mock"]`; all on a shared network; **LiteLLM published on loopback only ‚Äî `127.0.0.1:4000:4000`** (a dev-only unauthenticated gateway must not bind all interfaces; matches the `127.0.0.1` SDK base URLs), Ollama internal-only (no host port). Authored build-time-complete.
- **R2:** LiteLLM routes to Ollama via the native **`ollama_chat/<model>`** provider with `api_base: http://ollama:11434` AND the `OLLAMA_API_BASE` env set on the container; it serves the OpenAI **`/v1/chat/completions`** surface AND the Anthropic **unified `/v1/messages`** surface (not the raw `/anthropic` passthrough), plus the OpenAI **`/v1/embeddings`** surface when embeddings are opted in (R12; embeddings are OpenAI-surface only ‚Äî Anthropic has no embeddings endpoint). **Client-facing model contract (resolved in review):** base-URL repointing alone is NOT sufficient ‚Äî LiteLLM only serves models in `model_list`, so a **wildcard chat route** (`model_name: "*"` ‚Üí `ollama_chat/@@LLM_MODEL@@`) is configured so unmodified app code sending ANY chat model name (`gpt-4o`, `claude-sonnet-4-6`, ‚Ä¶) is transparently served by the local model on both chat surfaces (the explicit `local` alias is also kept). Embeddings route via the explicit **`local-embed`** alias (wildcard-across-endpoints is unreliable for `/v1/embeddings`); `build-config` emits `OPENAI_EMBEDDING_MODEL=local-embed` (opt-in only) so embedding callers have the name. Manual/integration smoke: both chat surfaces return a completion against the pulled model **using a real-looking model name (e.g. `gpt-4o`) to exercise the wildcard**.
- **R3:** A **deterministic mock mode** (LiteLLM `config.mock.yaml` with `mock_response`) runs under `--profile ai-mock` with **no Ollama container started**, returning canned output through **BOTH** the OpenAI `/v1/chat/completions` AND the Anthropic `/v1/messages` surfaces (verified by the optional Docker-enabled integration job in R10 ‚Äî LiteLLM only, no model; distinct from the no-daemon unit gate). Mode is selected by compose profile only ‚Äî no app-code edits, identical base URL. CI uses `ai-mock`. The mock is **a single editable canned `mock_response`** (in `config.mock.yaml`), not per-model/per-endpoint. The Anthropic-surface tool-use/streaming fidelity gaps are documented. **Scope note on the mock tool-use test:** a canned `mock_response` short-circuits the backend, so it CANNOT emit structured OpenAI `tool_calls` / Anthropic `content[].type=="tool_use"` ‚Äî the `ai-mock` integration assertion is therefore **"a tools-bearing request is tolerated ‚Äî no 5xx, well-formed response on both surfaces"** (a request-parsing/endpoint regression signal), NOT a structured tool-use round-trip. **Structured tool-use translation fidelity is a documented MANUAL real-inference check** (`--profile ai`), never asserted in CI.
- **R4:** The model is chosen **interactively at scaffold time, only when the user opts into installing the stack**. The menu offers: lightweight (e.g. `llama3.2:3b`), more powerful (e.g. `qwen2.5:7b`), abliterated (e.g. `huihui_ai/llama3.2-abliterate`, with a quality/guardrail/license caveat), and **"something else"** (skill guides selection by size/VRAM/task/license). The chosen model name is written to **`config.json` ‚Üí `localLlm.model`** (the single source of truth); `build-config` stamps it into the gitignored `litellm/config.yaml` (`ollama_chat/@@LLM_MODEL@@` ‚Üí raw model), and `system.sh up` exports `LLM_MODEL` from `config.json` so the model-pull container pulls it on first boot (no generated `.env`). The chosen/entered model (incl. the "something else" path) is validated against `^[A-Za-z0-9._/-]+(:[A-Za-z0-9._-]+)?$` at selection time. No hardcoded default model lives in committed templates. **After the chat-model prompt, a second opt-in prompt asks whether the project needs embeddings (R12)**; if yes, an embedding model is chosen (default `nomic-embed-text` + "something else", same grammar validation) and written to `config.json` ‚Üí `localLlm.embeddingModel`.
- **R5:** The `etc/local-llm/` stack + the local base-URL repoint are **opt-in and removable**: not laid down / not repointed unless the user opts in at scaffold time. **Inclusion mechanism (resolved in review):** the stack templates are authored under a **non-default-copied optional subtree `templates/_optional/local-llm/`** (the `_`-prefix keeps them out of the wholesale `templates/` copy, mirroring fn-2's `_`-private convention) ‚Äî so they NEVER appear in a default scaffold regardless of task ordering; the scaffold engine copies `_optional/local-llm/` ‚Üí the project's `etc/local-llm/` ONLY on `--local-llm` (fn-3‚Ä¶.4). This decouples the proof-point task (.1, authors the templates) from the opt-in wiring (.4). even when installed, the stack sits behind the `ai`/`ai-mock` compose profiles and is **excluded from the default `system.sh up`**. (`services.claude-api`/`services.openai-api` themselves are always present ‚Äî see R6.) Marked removable like `SampleApp`.
- **R6:** `config.json` and `config.deploy.json` **always** carry both `claude-api` and `openai-api` `services{}` entries, each with explicit `base_url` + `api_key` fields (key-set parity). Non-opt-in local values: `claude-api.base_url=https://api.anthropic.com`, `openai-api.base_url=https://api.openai.com/v1`, keys `REPLACE_ME`. Opt-in local values: `claude-api.base_url=http://127.0.0.1:4000`, `openai-api.base_url=http://127.0.0.1:4000/v1`, keys `sk-local-mock`. `config.deploy.json` uses `{{CLAUDE_API_BASE_URL}}`/`{{CLAUDE_API_KEY}}`/`{{OPENAI_API_BASE_URL}}`/`{{OPENAI_API_KEY}}` placeholders. `build-config` (unconditionally, no opt-in branch) distributes `ANTHROPIC_BASE_URL`/`ANTHROPIC_API_KEY` + `OPENAI_BASE_URL`/`OPENAI_API_KEY` into the consuming component's `.env` (the `Api` by default); external creds exempt from URL-safe validation. **When embeddings are opted in**, it ALSO emits `OPENAI_EMBEDDING_MODEL=local-embed` (the explicit embeddings alias ‚Äî see R2); chat callers need no model env var (the wildcard route serves any name). `build-config` **validates each `base_url`** against a concrete service-base-url grammar ‚Äî a restrictive host class (`[A-Za-z0-9.-]+` ‚Äî no IPv6 needed; `?#@[]:` and other non-host chars are rejected **in the host portion**): `^https?://[A-Za-z0-9.-]+(:[0-9]{1,5})?(/[^[:space:]]*)?$` (an optional path may follow ‚Äî query/fragment in a path are permitted, since a legitimate base URL can carry a path; this is an **injection-resistant host character-whitelist + port-range check, NOT full RFC-1123 hostname validation** ‚Äî it deliberately accepts odd-but-harmless hosts like `.example.com` or `-bad` (they'd simply fail to resolve);
