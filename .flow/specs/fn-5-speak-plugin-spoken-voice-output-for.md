## Architecture & Data Models
<!-- Source-tag breakdown: ~35% [user] / ~45% [paraphrase] / ~20% [inferred] -->

**One CLI, two transports.** A single `speak` CLI is the primitive everything calls — the Stop hook, the slash commands, and the user in a terminal. [paraphrase] It auto-detects **context** separately from **dependency availability**: Darwin → local host (a missing/broken `say` is reported as a missing *local* dependency, NOT a silent switch to forwarding); a **detected container** (Linux + a container marker like `/.dockerenv` / `/run/.containerenv`, or explicit `SPEAK_MODE=forward`) → forward to a host listener; **any other host (e.g. bare Linux, not a container)** → an "unsupported host OS" diagnostic, NOT a silent forward attempt against `host.docker.internal`. [user] This matches the two supported scenarios exactly — host = Mac, container = Linux. [paraphrase]

**Forwarding protocol.** The container-side CLI base64-encodes the cleaned text and sends it as a single line over a raw TCP connection via `nc`; the message also carries a session identifier. [paraphrase] The host listener (`speak --serve`) decodes each line and pipes it to `say`. base64 + one-line framing avoids HTTP parsing in bash 3.2; the encoder strips newlines (GNU `base64` wraps at 76 cols by default) so the frame stays exactly one line. [inferred]

**Listener.** `speak --serve` binds `127.0.0.1:8765` (override `SPEAK_PORT`), accepts connections in a respawn loop with a fast accept→enqueue path that hands off to a **single playback worker** draining a **bounded spool queue** (a temp dir of sequentially-named files — atomic enqueue, drop-oldest beyond a cap, **default 50**; NOT a named pipe, which can't drop-oldest and can block writers), so the accept path never blocks on `say` and concurrent arrivals queue rather than overlap. [user/paraphrase] Each accepted connection is read under a **shell watchdog** (bash 3.2 has no `coproc`/`timeout`): the **accept path itself is bounded** (not `nc … > file` which would let disk grow unbounded during the window): `nc -l …` is piped through a cap+1-byte reader/helper (e.g. `head -c $((MAX+1))` / a bounded `dd`) that stops at the cap, and the concrete `nc` PID/process group is killed + reaped once the cap is exceeded or no newline arrives — so an oversized/newline-less stream is never fully buffered to memory OR disk. A small helper is acceptable if pure bash proves too brittle. A stalled or newline-less client thus cannot wedge the accept loop or leave a zombie listener. Concurrent enqueue / prune / counter / session-log updates are guarded by an **atomic `mkdir` lock dir** (with trap cleanup); spool entries use atomic `mktemp` names with a sortable key so there are no duplicate or reordered sequence numbers and no file is pruned while being processed. [inferred] It shows a live status line: current state, the **active-session count** (distinct session ids seen within a recent rolling window, **default 60s**), and running success/failure totals (which are **playback outcomes only** — malformed/dropped/probe events are logged, not counted as failures). These counters and the session log are **file-backed** — bash background workers/subshells can't propagate variables to the parent, so the playback worker writes counts to state files the status line reads. [user] A malformed/undecodable payload is logged and skipped; the listener keeps serving. [user] A **listener-side max frame guard** uses a **bounded capture primitive**: under the watchdog it reads at most `SPEAK_MAX_FRAME_BYTES`+1 bytes of the line (**internal-only** named constant, fixed **65536** = 64 KiB; not a user-overridable env var) and **never buffers an unbounded newline-less frame** — a line over the cap, or with no newline within the cap, is rejected + logged *before* decode/enqueue. The cap is on the **encoded frame line** (the effective outer bound; the decoded payload is consequently ≤ ~48 KiB after base64 expansion). This stops a hostile/buggy client from causing memory pressure ahead of the count-based queue bound (client-side `SPEAK_MAX_CHARS` does not protect the listener). [inferred] The listener writes an **identity/pidfile** under the resolved data dir (`<data-dir>/listener.pid`) recording pid + script path + port (+ start time when available); host-side diagnostics treat it as *ours* only after a precise rule — **identity = pid alive + the live process command (via `ps`) contains `bin/speak` and `--serve` + the pidfile's recorded port equals the port being checked + the lifetime lock**; **health = identity + reachable** (reachability is reported separately and never folded into identity, so a respawn-gap unreachable never stales/deletes the pidfile) (NOT an exact script-path match — the listener may be launched as `./plugins/speak/bin/speak` or `${CLAUDE_PLUGIN_ROOT}/bin/speak`, which resolve differently; NO opaque token is required in `ps`). The pidfile schema is exactly `pid=<n>\nport=<n>\ncmd=<argv0>\nstarted=<epoch-or-unknown>`. Stale markers (pid dead / mismatch) are removed. **Double-start uses a lifetime lock, not just bind-failure:** the one-shot `nc -l` respawn loop leaves the port briefly unbound between accepts, so a second `speak --serve` could win the bind during that gap. `--serve` therefore acquires a `<listener_state_dir>/listener.lock` (atomic `mkdir`, with stale-lock validation against the pidfile) held for the whole listener lifetime and **checks the validated pidfile/lock BEFORE binding**, not only on bind-failure. **Identity is separate from reachability:** identity = pid alive + `ps` command shape + pidfile port + the lifetime lock; reachability is reported separately (with retry/backoff) and a momentary unreachable during the respawn gap never deletes the pidfile/marker or marks the listener stale. Container-side reachability stays TCP-level (port reachable) unless an ack protocol is added — documented. Malformed-frame and dropped (over-cap / queue-evicted) events are written to `<data-dir>/listener.log` (observable there, not folded into the status counters); the zero-byte `nc -z` probe is a **silent** health check — neither logged nor counted. [inferred]

**State on disk.** TWO named resolver helpers, used everywhere (no inline variants): (a) **`listener_state_dir` = `${SPEAK_DATA_DIR:-$HOME/.local/state/speak}`** (pid/log/spool) — **deliberately NO `CLAUDE_PLUGIN_DATA`** so a manually-started terminal listener and a **same-host** `doctor --listener` agree on the same dir (otherwise a terminal listener is misclassified "port in use by another process"). **Pidfile-identity validation is host-local only:** from a Dev Container, `$HOME`/filesystem are container-side and cannot see the host pidfile, so container/forward `doctor` is **TCP-reachability only** (never claims pidfile identity) unless `SPEAK_DATA_DIR` is deliberately shared via a mount; (b) **`plugin_state_dir` = `${SPEAK_DATA_DIR:-${CLAUDE_PLUGIN_DATA:-}}`** (toggle + debounce) — nounset-safe under `set -u`; when it resolves **empty**, the toggle is "unavailable" (never a `set -u` abort), **NO HOME fallback** — the hook always has `${CLAUDE_PLUGIN_DATA}`, and the toggle commands must read/write the SAME dir or the terminal toggle wouldn't affect the hook. So terminal `speak on/off` outside Claude requires `SPEAK_DATA_DIR` (or `${CLAUDE_PLUGIN_DATA}`); with neither set it reports "toggle unavailable" rather than silently writing a HOME dir the hook never reads. This makes "unset data dir → OFF/unavailable" well-defined. [inferred] Auto-speak on/off is a single **global** flag file `<plugin_state_dir>/auto-speak` containing exactly `on\n` or `off\n`, written atomically (temp + rename) by `/speak:on`/`/speak:off` and read by the Stop hook; any other/absent/unreadable content reads as OFF. [user] The once-per-session "listener unreachable" notice is gated by a marker file keyed by the hook's `session_id`. [user] The Stop hook reads the assistant's last message from the transcript JSONL it is handed on stdin; `/speak:now` instead receives the last response piped in by Claude. [user/paraphrase]

**OS seam.** TTS playback is isolated behind a single backend function (`say` today) so Windows/Linux backends can be added later without touching the transport, hook, or command layers. [user]

- **C1 — Mode-detection precedence** (context ≠ dependency availability): (1) explicit `SPEAK_MODE` wins — exactly `forward` or `local`, honored even on Darwin; any other non-empty value → clear CLI/`doctor` failure and, with auto-speak on, a debounced `invalid-config` notice; `SPEAK_MODE=local` on non-Darwin → "unsupported local TTS backend" (not forward, not generic missing-`say`). (2) else Darwin → local (missing/broken `say` = missing *local* dependency, never a switch to forward). (3) else Linux + container marker (`/.dockerenv` / `/run/.containerenv`) → forward. (4) else bare non-Darwin → "unsupported host OS" diagnostic (never a silent `host.docker.internal` attempt).
- **C2 — Wire frame + size cap:** exactly `<sanitized-session-id>\t<base64-cleaned-text>\n` — one tab, one newline-terminated line; base64 interior newlines stripped (GNU wraps at 76 cols). Listener rejects (log + skip, keep serving): zero or >1 tab, empty session id, empty payload, undecodable base64. Total encoded frame ≤ `SPEAK_MAX_FRAME_BYTES` = **65536** (internal named constant, NOT env-overridable; decoded payload consequently ≤ ~48 KiB). Sender pre-checks `len(session)+1(tab)+len(base64)+1(newline) ≤ 65536` and truncates cleaned text **by bytes on a UTF-8 character boundary** with a stderr/log notice (never spoken); the listener independently bounds its accept read at cap+1 bytes.
- **C3 — State-dir resolvers** (two named helpers, used everywhere, no inline variants): `listener_state_dir` = `${SPEAK_DATA_DIR:-$HOME/.local/state/speak}` (pidfile/lock/log/spool — deliberately NO `CLAUDE_PLUGIN_DATA`, so a terminal-started listener and same-host `doctor --listener` agree). `plugin_state_dir` = `${SPEAK_DATA_DIR:-${CLAUDE_PLUGIN_DATA:-}}` (toggle + debounce — nounset-safe; resolves empty → toggle "unavailable", NO HOME fallback). Pidfile-identity validation is host-local only; container/forward `doctor` is TCP-reachability only (unless `SPEAK_DATA_DIR` is deliberately mount-shared).
- **C4 — Reachability (two strengths, two targets):** the single fast `nc -z` probe (detected `-G1`/`-w1`) is ONLY for the hook's advisory notice decision — never a send gate (probe-then-send races the respawn loop; the send itself uses bounded retry/backoff and always dispatches when deps+port are valid). `doctor`/health checks use bounded retry/backoff so the one-shot `nc -l` respawn gap can't false-fail. Target = `host.docker.internal:$SPEAK_PORT` in forward/client context; `127.0.0.1:$SPEAK_PORT` in host/listener context. Reachability is always reported separately from identity — never folded in; a momentary unreachable never stales/deletes identity markers.
- **C5 — Doctor exit + classification:** exit non-zero ONLY on a tool/capability required for the current mode — local → `say` (listener reported "not needed", never fails; `jq` reported "required for the Stop hook" but not fatal); forward → `base64`/usable `nc` + an unreachable listener; listener → `say`/`base64`/`nc`; hook/`--hook` → `jq` + the detected-mode playback path (readiness-if-enabled; toggle state incl. OFF shown separately, never gating). `doctor --listener` is host-only, four-way: reachable ∧ identity → healthy (0); reachable ∧ ¬identity (post stale-cleanup) → "port reachable but not recognized as this listener" (non-0); ¬reachable ∧ no valid pidfile/lock → "unrelated process / nothing on port" (non-0); live our-listener recorded on a DIFFERENT port than the one checked (C7 rc 11) → "listener running on :recorded, not :checked" (non-0, markers preserved); in container/forward context it degrades to TCP-reachability only. Listen (`-l`) capability is NEVER pre-probed — proven only by `speak --serve` actually binding; `doctor` reports it "not checked; proven by --serve".
- **C6 — nc capability detection** (owned by `.2`; every consumer reuses it, no re-derivation): detect via option/usage-error probing with a **distinct unsupported-capability return code** never conflated with refused/unreachable — connect-timeout `-G` (BSD) else `-w` (GNU/OpenBSD); EOF-shutdown `-N` else `-q 0` (neither → fail with "install netcat-openbsd"); zero-I/O probe `-z`. No `nc -l` dependency in detection.
- **C7 — Listener identity + lifetime lock:** pidfile `<listener_state_dir>/listener.pid`, schema exactly `pid=<n>\nport=<n>\ncmd=<argv0>\nstarted=<epoch-or-unknown>`. **Identity** = pid alive + live `ps` command contains `bin/speak` AND `--serve` + recorded port == checked port + lifetime lock held (NO reachability, NO opaque token, NOT an exact script-path match). Lifetime lock = `<listener_state_dir>/listener.lock` (atomic `mkdir`, stale-validated against the pidfile), held for the whole listener lifetime, checked BEFORE binding. **Health** = identity + reachable (per C4). **Stale** = the marker no longer describes a live our-listener at all — pid dead, live command not a `speak --serve` process, or lock/pidfile mutually inconsistent → markers removed. A **live** our-listener whose recorded port differs from the port being checked is NOT stale (helper rc 11): markers are preserved, a second `--serve` on the different port refuses (one listener per state dir — starting would clobber the live listener's state), and diagnostics name the actual port. Hardening shipped by `.8` (review-driven, now contractual): a startup grace window (`SPEAK_LOCK_GRACE_SECS`) protects a concurrently-starting listener from stale-cleanup; the pidfile is published only AFTER bind confirmation; a pidfile write failure aborts `--serve` startup.
- **C8 — Debounce markers:** `<plugin_state_dir>/debounce/<sanitized-session-id>.<reason>`, reason ∈ `listener-unreachable` / `missing-jq` (fixed GLOBAL key — session id unavailable without `jq`) / `missing-<tool>` / `invalid-port` / `invalid-config`; stale markers pruned after 7 days on hook entry. `sanitize_session_id` (charset `[A-Za-z0-9._-]`, capped length; empty-after-sanitize → `unknown`/short hash) is applied before ANY use of a session id in a frame, log line, or filename — no raw id ever becomes a filesystem path.
- **C9 — Constants + env:** port default **8765** (`SPEAK_PORT`, validated integer 1..65535 by client, listener, and doctor); spool queue cap **50** (drop-oldest, logged); active-session rolling window **60s**; encoded-frame cap **65536** (C2); debounce prune age **7 days**; `SPEAK_MAX_CHARS` unset = no cap (invalid → treated unset + stderr diagnostic); Stop-hook `timeout` (hooks.json field, seconds) = **10**.

## Edge Cases & Constraints
<!-- Source-tag breakdown: ~30% [user] / ~70% [inferred] -->

- **bash 3.2 floor.** macOS ships bash 3.2.57; the CLI must run on that lower bar (no associative arrays, no `${var,,}`, no `mapfile`/`readarray`) so one script works host-side and container-side. [inferred]
- **No `/dev/tcp`.** Apple's bash has `/dev/tcp` disabled, so transport and reachability checks go through `nc`. No `timeout`/`gtimeout` on the host, so reachability uses `nc`'s own connect-timeout flags (`-z -G1`). [inferred]
- **nc flavor asymmetry → capability detection.** Host is BSD `nc`; the container may be GNU, OpenBSD, or nmap `nc` with divergent flags. Do NOT hard-code `-G`/`-N`. Detect: connect-timeout via `-G` (BSD) else `-w` (GNU/OpenBSD); EOF-shutdown via `-N` else `-q 0`; if neither send-shutdown flag works, fail with clear "install netcat-openbsd" guidance. The host listener uses a `while :; do nc -l ...; done` respawn loop (BSD `nc -l` serves one connection then exits). [inferred]
- **base64 framing + decode flags.** GNU `base64` wraps at 76 cols by default; the encoder must strip newlines (e.g. `base64 | tr -d '\r\n'`) so a long utterance stays exactly one line. Decode flags also differ — BSD `base64 -D` vs GNU `-d` — so detect the working decode flag rather than hard-coding. Round-trip is tested with payloads >76 chars. [inferred]
- **Session-id sanitization.** `SPEAK_SESSION` is user-controlled; sanitize the session id to a safe charset (`[A-Za-z0-9._-]`, capped length) before framing/logging so it can't break the line protocol or pollute logs. [inferred]
- **Dependencies (per context).** Required tools differ by context: **local CLI** → `say` (base64 is wire-transport only, not needed for local playback); **forward client (container)** → `base64`, `nc` (+ `jq` when invoked by the hook); **listener (host)** → `say`, `base64`, `nc`; **Stop hook** → `jq` (+ the CLI). So `jq` is needed on the **host** too (the hook parses transcript JSON there), not just the container. If a required-in-context tool is missing the plugin degrades gracefully and the notice names what to install (no hard crash). The preferred netcat is **`netcat-openbsd`** — generic `netcat` packages may lack the required connect-timeout/EOF-shutdown/`-z` behavior; docs name it explicitly. **Provisioning:** a project using speak from a Dev Container must install `netcat-openbsd` (+ `jq`) in its `.devcontainer/setup.sh` per the repo dev-container standard; the README + `dev-container.md` state this (the plugin does not auto-provision — there's no `.devcontainer` in this plugins repo itself). [user]
- **`${CLAUDE_PLUGIN_DATA}` unset.** Outside Claude (e.g. terminal `speak doctor`/`speak status`), the data dir may be unset; the toggle is then treated as OFF/unavailable and reported by `doctor` rather than crashing. [inferred]
- **Listener down — auto vs explicit.** When the Stop hook can't reach the listener, the notice is non-blocking and debounced to once per session. An explicit `/speak:now` against an unreachable listener instead shows an immediate, clear message (bypasses the debounce). [user]
- **Transcript extraction (exact algorithm).** Inspect only the FINAL transcript record: speak iff it is an assistant message, extracting its text parts (skip tool_use/thinking). If the final record is not an assistant message, or its text is empty after cleaning, speak nothing and exit 0. **Never scan backward** past a non-assistant final record — that would replay a stale prior response. [inferred]
- **Double-start.** Running `speak --serve` when the port is already held exits cleanly with a friendly "already running on 8765" message rather than a raw `nc` error. [user]
- **Security.** Listener binds loopback only and treats incoming text as data (passed to `say` via stdin, never interpolated into a shell) — no command injection. [inferred]
- **Length.** Full cleaned response is spoken by default; `SPEAK_MAX_CHARS`, when set to a positive integer, truncates with a brief notice; an invalid value (zero/negative/non-numeric) is treated as unset (no cap) with a diagnostic to stderr. [user]
- **macOS firewall.** First bind may trigger a macOS "allow incoming connections" prompt (documented; loopback bind minimizes it). [inferred]
- **Remote hosts.** `host.docker.internal` resolves to the Docker host; remote/Codespaces containers point at a machine that isn't the user's Mac — out of scope (see Boundaries). [paraphrase]

## Acceptance Criteria
<!-- scope: both -->

- **R1:** On a macOS host, the `speak` CLI speaks supplied text aloud via the OS TTS voice. [user]
- **R2:** Spoken output is the "cleaned" response — markdown, code blocks, and formatting are stripped before TTS. [user]
- **R3:** A `Stop` hook automatically speaks each assistant response when auto-speak is enabled. It does preflight/classification synchronously — **listener reachability is checked only in `forward` mode; in local mode it preflights only `say` (+ `jq` for the hook's own parsing — `base64` is wire-transport only, not a local dep), never asking the user to start `speak --serve`** — then dispatches playback **asynchronously** (backgrounded, stdout/stderr redirected to a temp/log) and returns immediately — `say` is synchronous, so a blocking call would stall the hook for the whole speech duration and risk the hook timeout; this preserves the non-blocking contract. [user]
- **R4:** Auto-speak defaults to OFF; `/speak:on` enables and `/speak:off` disables it, persisted so the Stop hook reads the current state. [user]
- **R5:** `/speak:now` asks the user which recent assistant response to read (a short pick list, most-recent first — several assistant responses can accumulate between user prompts) and speaks the chosen one, regardless of the toggle. Only the explicit `/speak:now` path offers this selection; auto-speak stays limited to the just-finished response. [user]
- **R6:** When Claude runs in a Dev Container, the CLI forwards spoken text to a host listener so audio plays on the host. [user]
- **R7:** The host listener is started manually by the user (`speak --serve`); the plugin never auto-starts it. [user]
- **R8:** When the hook cannot reach the listener from the container, it surfaces a non-blocking text notice containing the workspace-relative command to start it (e.g. `./plugins/speak/bin/speak --serve`). [user]
- **R9:** That unreachable-listener notice is debounced — shown at most once per session. [user]
- **R10:** On a host (no container), the CLI plays audio directly with no listener required. [paraphrase]
- **R11:** OS-specific TTS is isolated so macOS works now and Windows can be added later without rearchitecting. [user]
- **R12:** The plugin is registered in the marketplace at `./plugins/speak`. [user]
- **R13:** `/speak:status` reports whether auto-speak is on and whether the listener is reachable. [inferred]
- **R14:** The forwarding transport is base64-encoded text with newlines stripped so each utterance is exactly one line (verified for payloads >76 chars, since GNU base64 wraps), over a raw TCP connection via `nc`; each message carries a session identifier. [paraphrase]

[user]

- **R19:** Cleaning drops fenced code blocks entirely, strips markdown syntax (headers/bullets/emphasis), drops URLs (keeping link text), and collapses whitespace. [paraphrase]

- **R23:** Required tools are detected per context (per the dependency matrix — **`say` local** (base64 is NOT needed for local `say` playback — it's wire-transport only), `base64`/`nc` forward client, `say`/`base64`/`nc` listener, `jq` hook). Default/forward `doctor` likewise reports `-l` "not checked in this context". A missing send/probe capability in forward mode is treated like a missing dependency. If any required-in-context tool/capability is missing the plugin degrades gracefully and the notice names what to install. [user]

Each dependency is labelled required-for-the-detected-mode vs optional/other-feature; `doctor` exits non-zero only when a tool **required for the current mode** is missing (local → `say`; forward → `base64`/`nc` + an unreachable listener; listener → `say`/`base64`/`nc`; hook → `jq`). Default `speak doctor` reports `jq` as "required for the Stop hook" but does NOT fail on a missing `jq` in local/forward mode. **`speak doctor --hook` reports readiness-if-enabled** — `jq` PLUS the detected-mode playback path (local → `say`; forward → `base64` + usable `nc` + listener reachable) — so it fails when auto-speak *would* fail if turned on; the current toggle state (incl. OFF) is shown separately and does not by itself make `--hook` pass. A healthy `speak "<text>"` install still passes default `/speak:status`. [user]
- **R25:** An explicit `/speak:now` against an unreachable listener shows an immediate clear message, bypassing the once-per-session debounce. [user]
- **R26:** The plugin ships bash tests for pure-logic units (cleaning, base64 round-trip, last-message handling, mode detection) plus a coverage script; `/speak:test` runs them. [user]
- **R27:** Starting `speak --serve` when a listener already holds the port exits cleanly with a friendly "already running" message (bind-failure + the listener pidfile distinguish *our* listener from an unrelated process on the port). [user]
- **R28:** The Stop hook's stdout is either empty or a single JSON object carrying `systemMessage` (the child `speak` invocation's stdout/stderr is captured, never leaked to the hook's stdout); the notice is non-blocking with exit 0. [paraphrase]
- **R29:** The command surface is exactly `/speak:now`, `/speak:on`, `/speak:off`, `/speak:status`, `/speak:test` (via command files `now.md`/`on.md`/`off.md`/`status.md`/`test.md` in the `speak` plugin's `commands/` dir — the `speak:` namespace comes from the plugin name), plus terminal `speak <text>`, `speak --serve`, `speak doctor` (with `speak status` as an alias), and `speak on`/`speak off` (the flag writes the slash toggles call). ## Boundaries
<!-- scope: business -->

- No auto-start / login-item / LaunchAgent for the listener in this iteration. [user]
- Non-macOS hosts are not implemented now (structure for Windows later, but don't build it). [user]
- Remote dev containers / Codespaces / remote-SSH are out of scope — audio only works when the container runs on the same machine as the speakers. [user]
- High-quality cloud TTS (ElevenLabs/OpenAI etc.) is not in scope; native OS voice only. [user]
- **Local-mode playback is not serialized (accepted limitation).** The no-overlap queue (R16) exists only in the forward-path listener. In local mode the Stop hook detaches `speak` → `say` directly, and macOS runs concurrent `say` processes simultaneously — back-to-back responses (or an explicit `/speak:now` during auto-speech) can produce overlapping audio. Accepted for this iteration; revisit only if it proves annoying in practice. [user]
- **No stop/interrupt for in-flight speech (accepted limitation).** `/speak:off` disables future auto-speak but does not kill an utterance already playing; there is no kill-speech command in this iteration. [user]

## Decision Context
<!-- scope: both -->

- **TTS engine:** native OS voices (macOS `say`) chosen over a cloud API — free, offline, no keys. [user]
- **Trigger model:** auto-speak via Stop hook with an on/off toggle, defaulting to OFF (opt-in). [user]
- **Listener lifecycle:** manual `speak --serve` rather than auto-start, because a container cannot (and should not) launch a process on the host — that isolation boundary is deliberate. [paraphrase] The detect-and-instruct hook is the safety net. [user]
- **Host-path in the hook notice:** use the workspace-relative path, not a full physical path — "not worth the added complexity." [user]
- **Windows:** deferred but designed-for via isolated OS seams. [user]
- **Transport:** raw base64 line over `nc` chosen over hand-rolled HTTP — robust in bash 3.2, newline-safe, and we own both ends. [paraphrase]
- **Concurrency:** a bounded ordered queue chosen over drop-if-busy or interrupt — ordering is preserved and nothing is lost until the configured cap is exceeded, at which point the oldest entries are dropped and logged. [paraphrase]
- **Voice config:** system default voice only (no env knobs) — change it in System Settings if desired. [user]
- **Toggle scope:** global rather than per-project — voice is a personal "on right now" preference. [user]
- **`/speak:now` text source + selection:** Claude pipes the chosen message (best-effort verbatim) rather than the CLI re-locating the transcript — simpler for the on-demand path, and keeps the terminal command surface exact (no public `speak --last`; any transcript-reuse stays a private/sourceable helper). [user] Because several assistant responses can accumulate between user prompts, `/speak:now` first asks which recent response to read (a model-mediated pick list, most-recent first — cheap because Claude already holds the conversation in context, no transcript parsing); auto-speak never asks and speaks only the just-finished response. [user]
- **Exit codes:** simple 0/1 rather than the repo's 64/127 contract — the user's explicit call for this CLI. [user]
- **Command namespace:** the five commands are packaged under the `speak` plugin namespace (`/speak:now`, `/speak:on`, `/speak:off`, `/speak:status`, `/speak:test`) instead of flat `/speak-*` names — cleaner grouping, and the namespace comes free from Claude Code's `/<plugin>:<command-file>` convention (no dispatcher work). `/speak` itself became `/speak:now` because a plugin command can't be the bare plugin name. [user]

## Plan: gap resolutions

Resolutions for the gaps surfaced during planning (defaults chosen; revisit if the early proof point invalidates them):

- **Session-id source:** Stop hook uses `session_id` from its stdin JSON; the terminal `speak <text>` path uses `${SPEAK_SESSION:-$(hostname)}`. The listener's active-session count is the number of distinct session ids seen in a recent rolling window (informational).
- **Mode detection (exact precedence):** (1) explicit **`SPEAK_MODE`** wins first — exactly `forward` or `local` (honored even on Darwin; any other non-empty value fails clearly in the CLI/`doctor` and, with auto-speak on, yields a debounced `invalid-config` hook notice). `SPEAK_MODE=local` selects local-playback mode but the only implemented backend is Darwin `say`, so **non-Darwin forced-local reports "unsupported local TTS backend"** (not forward, not a generic missing-`say`); (2) else Darwin → local; (3) else Linux + container marker (`/.dockerenv`/`/run/.containerenv`) → forward; (4) else → "unsupported host OS" diagnostic (no silent forward). This precedence is defined once as **C1** in §Canonical Contracts; tasks cite C1 rather than restating it.
- **Reachability — two strengths:** a single fast `nc -z` probe is used **only** for the hook's advisory notify decision; `doctor`/`doctor --listener`/health reporting use a **bounded retry/backoff** reachability check so the one-shot `nc -l` respawn gap doesn't cause a false failure. Both target `host.docker.internal:${SPEAK_PORT}` in **forward/client** context and `127.0.0.1:${SPEAK_PORT}` in **listener/host** context (`speak --serve` preflight, `speak doctor --listener`) — using the wrong target would give false-unreachable diagnostics on the host.
- **Forward-send reliability:** the listener is a one-shot `nc -l` respawn loop, so a `nc -z` probe consumes one accept cycle and a probe-then-send can race the respawn. The actual forward send therefore uses the detected connect-timeout + EOF-shutdown flags with **bounded retry/backoff** (a few attempts) and IS the reliable delivery path; the `-z` probe is advisory only, never a hard gate that drops a send. **In the hook (forward mode):** when deps + port are valid it ALWAYS dispatches the async send-with-retry; the probe result only decides whether to *also* emit the debounced unreachable notice — a transient false-negative probe never suppresses the send. Terminal forward `speak "<text>"` with the listener down fails fast and clearly — it never hangs.
- **Transcript extraction:** inspect only the FINAL transcript record — speak iff it is an assistant message (extract text parts, skip tool_use/thinking); if it is not assistant, or empty after cleaning, exit 0 and speak nothing. Never scan backward past a non-assistant final record.
- **Debounce markers (by session + reason):** markers live under `<plugin_state_dir>/debounce/<sanitized-session-id>.<reason>` (reason ∈ `listener-unreachable` / `missing-jq` / `missing-<tool>` / `invalid-port` / `invalid-config`); stale markers older than **7 days** are pruned on hook entry so a distinct actionable notice isn't suppressed by an unrelated earlier one; the `missing-jq` fallback key stays deterministic. Prune entries older than a fixed age on hook entry (best-effort, bounded growth).
- **Queue bounding + shutdown:** the ordered spool queue is bounded (drop-oldest beyond a cap, logged); a `trap` on INT/TERM stops accepting and removes the spool temp dir (no named pipe is used).
- **Toggle read-failure:** a missing or unreadable flag file means OFF (safe default).
- **`say` failure** (no audio device / removed voice): counted in the listener's failure total and logged; never crashes the listener.
- **`/speak:test` contract:** runs `coverage.sh --plain` (plain mode = skip kcov, fast + deterministic); exits 0 on all-pass, non-zero on any failure.
- **Hook path resolution:** the Stop hook invokes `"${CLAUDE_PLUGIN_ROOT}/bin/speak"` explicitly (env var is exported to hook processes) rather than relying on PATH.
- **`speak doctor` exit:** 0 when healthy; non-zero on a missing required dependency, or — in **forward mode only** — an unreachable listener. In **local mode** the listener is reported "not needed" and never causes a failure (a healthy host install must pass diagnostics without `speak --serve` running) (R7/R10). Listener-context dependency checks (say/base64/nc) run as a **preflight inside `speak --serve` before binding** (reported there); a `speak doctor --listener` flag may expose the same check without growing the slash-command surface.
- **jq-less hook order (exact):** the hook reads the toggle flag from `${CLAUDE_PLUGIN_DATA}` FIRST (a plain file read — no `jq` needed). If auto-speak is on and `jq` is absent, it still honors the loop guard with a tiny `jq`-free check — read stdin once into a variable/temp, then match the **whitespace-tolerant** regex `"stop_hook_active"[[:space:]]*:[[:space:]]*true` (handles minified `"stop_hook_active":true`); if it matches, exit 0 silently (no notice). All hook `systemMessage` output goes through **one shared JSON emitter**: with `jq` present, `jq -n --arg message ...` (safe encoding of any value, incl. an invalid `SPEAK_PORT` containing quotes/newlines); on the no-`jq` path, hard-coded JSON-safe static strings only (no untrusted interpolation). Either way hook stdout is always valid JSON per R28 — never string-interpolated. Otherwise emit at most one `systemMessage` keyed by the fixed `missing-jq` debounce marker (or exit 0 silently) and exit 0. `session_id`/`transcript_path` are not read on this degraded path.
- **base64 framing:** the encoder strips newlines so a long utterance is exactly one line; round-trip is tested with payloads >76 chars (GNU base64 wraps). Decode-flag portability (BSD `-D` vs GNU `-d`) is detected, not hard-coded.
- **nc capability detection:** probe flags rather than assume — connect-timeout via `-G` (BSD) else `-w` (GNU/OpenBSD); EOF-shutdown via `-N` else `-q 0`; if neither shutdown flag exists, fail with "install netcat-openbsd" guidance. `doctor` verifies the local `nc` can do both.
- **jq everywhere + missing-dependency notice:** `jq` is required on the host too (the Stop hook parses transcript JSON there), not just the container; `doctor` checks it on whichever side it runs. More generally: when auto-speak is enabled and **any** required-in-context dependency or transport capability is missing (`jq`/`say`/`base64`/`nc`/a usable nc-shutdown flag), the hook emits **one debounced `systemMessage`** with **reason-specific content** — listener-unreachable → the `./plugins/speak/bin/speak --serve` command; missing dependency/capability → names the exact tool/capability + install guidance (NOT the `speak --serve` command, which wouldn't help that case); an invalid `SPEAK_PORT` → an `invalid-port` notice naming the bad value + the valid 1..65535 range — never hard-fails (no `set -e` abort), keeps stdout otherwise empty, and always exits 0. Without `jq` the debounce key falls back to a fixed `missing-jq` key (session_id can't be parsed).
- **Session-id sanitization:** `SPEAK_SESSION` (and the hook's `session_id`) is user/host-controlled, so a single shared `sanitize_session_id` helper (charset `[A-Za-z0-9._-]`, capped length; or a hash) is applied before it is used in a wire frame, in logs, **or as a debounce-marker filename** (`<plugin_state_dir>/debounce/<sanitized-id>.<reason>` per C8). No raw session id ever becomes a filesystem path — prevents path traversal. If sanitization yields an empty string (e.g. `SPEAK_SESSION='///'`), fall back to a safe non-empty value (`unknown` or a short hash) so a frame never has an empty session id.
- **`${CLAUDE_PLUGIN_DATA}` unset:** outside Claude the data dir may be unset → toggle treated as OFF/unavailable; `doctor` reports it, never crashes.
- **Hook stdout contract:** the hook prints nothing, or exactly one JSON object with `systemMessage`; the child `speak` invocation's stdout/stderr is captured so only intentional hook JSON reaches stdout.
