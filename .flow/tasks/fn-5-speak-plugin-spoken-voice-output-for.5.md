## Approach

- `commands/*.md` slash-command files (markdown). `/speak`: the command file instructs Claude to send the immediately-previous assistant message body **as faithfully as possible** (best-effort verbatim â€” slash commands are model-mediated, so byte-identity isn't guaranteed for long markdown/code fences) to `"${CLAUDE_PLUGIN_ROOT}/bin/speak"` via a **robust stdin handoff** (a temp file, or a quoted heredoc with a generated unlikely delimiter â€” raw assistant markdown can contain backticks, code fences, and arbitrary delimiters). Do NOT add a public `speak --last` terminal command (R29 fixes the surface); any transcript-reuse stays a private/sourceable helper only. When forwarding is unreachable, show an **immediate** clear message (bypass the once-per-session debounce, since the user asked explicitly). `/speak-on` â†’ `speak on`; `/speak-off` â†’ `speak off`; `/speak-status` â†’ `speak doctor`.
- `doctor` subcommand (with `speak status` as an alias): full diagnostic â€” toggle state (treating an unset `${CLAUDE_PLUGIN_DATA}` as OFF/unavailable rather than crashing), detected mode (local/forward), port, listener reachability (round-trip using the detected connect-timeout flag `-G1`/`-w1` + `-z`), and presence of the TTS backend **`say`** (in local/listener contexts) plus `nc`/`base64`/`jq` (incl. `jq` on the host) â€” for `nc`, check the capability set (connect-timeout `-G`/`-w`, EOF-shutdown `-N`/`-q0`, zero-I/O probe `-z`), not mere presence; **listen `-l` is NOT pre-probed** (portable nc/bash can't bind ephemeral port 0) â€” default/forward `doctor` reports `-l` as "not checked in this context", and it's proven only by `speak --serve` binding or `speak doctor --listener`. In forward mode a missing send/probe capability fails the diagnostic. Resolve the listener pidfile lookup via **`listener_state_dir` = `${SPEAK_DATA_DIR:-$HOME/.local/state/speak}`** (NOT `CLAUDE_PLUGIN_DATA` â€” matches the terminal listener); toggle reads via `plugin_state_dir` = `${SPEAK_DATA_DIR:-${CLAUDE_PLUGIN_DATA:-}}` (nounset-safe; empty â†’ "toggle unavailable"). `speak doctor --listener` does NOT bind â€” it checks listener-mode deps (`say`/`base64`/`nc`) + existing-listener identity (pidfile) + reachability to the configured port; `speak --serve` binding is the ONLY authoritative listen check (NOT `doctor --listener`); `doctor --listener` explicitly reports listen capability "not checked; proven by --serve", and **exits non-zero when** any listener dep is missing, identity is false, reachability is false, or an unrelated process owns the port (exit 0 only for a healthy reachable *our* listener). Identity = pid + ps-shape + pidfile-port + lifetime-lock; **health = identity + reachable** (reachability reported separately, never folded into identity). **`doctor --listener`: listener identity/dependency validation is host-local only; in forward/container context it degrades to TCP reachability only.** On the host it checks deps + identity + reachability and classifies: `reachable&&identity`â†’healthy(0); `reachable&&!identity` (post stale-cleanup)â†’"reachable but not our listener"(non-0); `!reachable&&no valid pidfile/lock`â†’"unrelated/nothing on port"(non-0). **In a container/forward context it is TCP-reachability only â€” NO listener-dep or pidfile-identity checks** (`$HOME`/fs are container-side and don't describe the host listener), unless `SPEAK_DATA_DIR` is shared via a mount. **`speak doctor --hook` reports readiness-if-enabled**: `jq` PLUS the detected-mode playback path (local â†’ `say`; forward â†’ `base64`+usable `nc`+listener reachable), failing when auto-speak *would* fail if turned on; the current toggle state (incl. OFF) is shown separately and does not by itself make `--hook` pass. Label each dependency **required-for-the-detected-mode vs optional/other-feature**, and exit non-zero ONLY when a tool required for the current mode is missing: local â†’ `say` (base64 is wire-transport only, NOT a local dep); forward â†’ `base64`/`nc` (+ an unreachable listener); listener â†’ `say`/`base64`/`nc`; hook â†’ `jq`. In **local mode** the listener is reported "not needed" and never fails (a healthy host install must pass without `speak --serve` running); `jq` is reported as "required for the Stop hook" but does NOT fail default `speak doctor` in local/forward mode â€” only the hook context (or `speak doctor --hook`) treats missing `jq` as a failure, so a healthy `speak "<text>"` install passes `/speak-status`. Validate and report `SPEAK_PORT` (1..65535). A `speak doctor --listener` flag runs the listener-context check on the host without adding a slash command; reachability uses a **bounded retry/backoff** check (not the single fast hook probe) **against `127.0.0.1`** in host/listener context (not `host.docker.internal`, which would false-fail on the Mac host) so the one-shot `nc -l` respawn gap doesn't false-fail, and host-side it also confirms it's *our* listener via the pidfile â€” identity = pid alive + live `ps` command contains `bin/speak` and `--serve` + the pidfile's recorded port == the checked port + lifetime lock (**NO reachability folded into identity** â€” reachability reported separately; NO opaque token in `ps`; NOT an exact script-path match, since `./plugins/speak/bin/speak` and `${CLAUDE_PLUGIN_ROOT}/bin/speak` differ); pidfile schema `pid=â€¦\nport=â€¦\ncmd=â€¦\nstarted=â€¦`; an open port held by an unrelated process is reported "port in use", not a healthy listener.
- Dependency detect-and-warn: the **shared helper is owned/implemented by `.4`** (hook preflight needs it first); this task **reuses** it for `doctor`'s diagnostic rendering + any listener-specific extensions â€” no duplicated detection logic. The single-source helper in `bin/speak` â€” no duplicated/divergent logic. `doctor` reuses the **shared `nc` capability helper from `.2`** (connect-timeout/EOF-shutdown/`-z` via usage-error probing, distinct unsupported-capability return, no `nc -l` dependency) â€” it does not re-derive detection. The unsupported-capability return is classified separately from connection result (refused/unreachable/DNS-fail), so "flag unsupported" is never conflated with "listener unreachable". Tests use fake `nc` stubs for the unsupported-flag vs refused-connection cases.
